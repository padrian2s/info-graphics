THESIS • “Disposable” is economics, not vibes

When software cost → ~0, the bottleneck becomes attention.

The current discourse often treats “disposable software” as a single thing. It isn’t. It splits into personal throwaway apps and disposable features inside products. Those two modes have different winners, different org shapes, and different failure modes.

INDUSTRY INVERSION

What changed: the cost curve flipped

For decades, software cost mostly meant engineering cost. If engineering is expensive, you plan heavily. When code generation becomes cheap, the planning layer can become overhead — but only in contexts where users tolerate variance.

Reading the chart: As AI leverage rises, code cost drops rapidly. Attention cost (goal-setting, reviewing, maintaining, building trust) stays stubbornly high.

Even if agents can generate massive codebases quickly, someone still must decide the goal, configure the run, review outcomes, and keep the system correct as reality changes.

Key distinction: generation vs direction ›
Generation collapsed. Directing attention toward a goal did not. When people say “software is free now,” they often mean “typing code is cheap now.” But choosing what to build, validating it, and owning it over time is still expensive.
Why this matters: disposability changes rituals ›
Heavy planning, alignment meetings, and long roadmaps were insurance against expensive mistakes. When features are cheap to try, insurance can become drag — but only where the customer accepts change.
DISPOSABLE ≠ ONE THING

Two phenomena wearing the same name

Click to switch modes. Each has different “what good looks like,” different risks, and different winners.

Rule of thumb: Personal disposability is democratization. Product-feature disposability is a strategy bet — it can win in developer markets and fail spectacularly in trust markets.
RITUALS THAT DIE

When software is cheap, “planning as insurance” changes

Not “no planning.” But many rituals grew around expensive software. Cheap iteration shifts the balance from predict → probe.

What becomes overhead in high-variance markets ›
Long design docs, heavyweight specs, alignment meetings, and fixed roadmaps can become less useful when you can ship and learn quickly. The “insurance” premium starts to look like waste.
What still matters everywhere: vision ›
A compelling product direction is not optional. Disposability changes execution cadence, not the need for taste, priorities, and clarity about what game you’re playing.
Structural tension: iteration vs stability ›
Maximum iteration speed demands churn. Stability demands continuity. You can’t fully maximize both at the same time; your customer decides which one pays.

WHO TOLERATES CHANGE?

The tolerance gap is structural

Disposability works when customers accept variance. It fails when customers are paying for peace of mind. Here’s a simple “tolerance to instability” sketch.

Why it matters: If even the most change-tolerant users complain at full speed, the strategy won’t generalize to buyers whose job is avoiding risk.
ENTERPRISE ≠ FEATURES

Enterprise buyers are buying reliability

They want software they can ignore. That’s why contracts come with SLAs, uptime promises, and “someone to call” when the system is down.

Why “we’ll all vibe-code our own Salesforce” breaks ›
The flaw isn’t code generation. It’s attention + maintenance. Building internal replacements pulls your best people away from higher-value work, then traps them in an ongoing security/maintenance loop on a non-core system.
Disposable features clash with trust markets ›
Finance and HR teams don’t want the interface rearranging itself every week. They want Tuesday to look like Monday, and next quarter to look like this quarter. Reliability is the product.
So what’s the AI edge for enterprise? ›
Not “a chatbot.” Reactive AI waits to be asked. The compelling edge is proactive capability — but only after reliability earns trust.

PROACTIVELY RELIABLE

The enterprise ladder: reliability → autonomy

Proactive agents create value users didn’t know they were missing — but only when trust is deep enough that users don’t feel forced to supervise every action.

1

Reliability baseline

Uptime, security hygiene, predictable UX, support. Trust comes first.

2

Suggest, don’t act

Agent surfaces insights and drafts actions — user approves. Build a track record.

3

Low-stakes autonomy

Autonomous actions where mistakes are reversible (drafts, alerts, safe updates).

4

Expand scope with guardrails

Policies, audit logs, explainable plans, rollbacks. Autonomy increases with proof.

5

Quiet competence

The agent just does the right thing. Users stop “checking” because it’s never wrong.

Interface hint: Simpler, stable surfaces (terminal, workflow “show your work”) absorb fast backend change without constantly re-teaching users.
